# 少年博文 MVP Design System v0.1

版本：v0.1  
来源：Figma 文件 `少年博文`，页面 `给开发`，节点 `66:4169`  
适用范围：移动端 MVP，一期核心场景为首页、详情页、个人中心。  
定位：这是少年博文最底层、最稳定的设计规范，用于指导后续人工设计、AI 生成页面、模块规格拆分和工程实现。

## 1. 产品定位

少年博文是面向青少年的新闻阅读与表达学习产品。它不是传统资讯客户端，也不是重课程型学习平台，而是把真实新闻转化为适合青少年理解、讨论和写作积累的内容工具。

核心价值：

- 帮助青少年看懂新闻背后的影响链路。
- 将复杂公共议题转化为可理解、可表达、可讨论的内容。
- 用音频、考点、金句、观点、中英对照等模块降低阅读门槛。
- 让用户在阅读、评论、收藏、生成简报等行为中积累表达素材。

## 2. 目标人群

主要用户：

- 10-18 岁青少年，包含不同阅读理解能力和表达能力阶段。
- 关注语文写作、时事理解、英语阅读、表达训练的学生。

间接用户：

- 家长：关注内容安全、价值导向、学习收益。
- 老师或教育从业者：关注议题解释、写作素材、课堂讨论价值。

年龄段内容策略：

- `10岁以下`：更短、更直观，用生活化比喻解释新闻影响。
- `11-14岁`：保留背景和因果链路，适合初中阶段理解和表达。
- `15岁以上`：信息密度更高，可引入概念、争议、观点和长文阅读。

## 3. 视觉风格指引

整体气质：

- 可信但不严肃。
- 清爽但不空洞。
- 有学习感但不教材化。
- 面向青少年但不幼稚。

视觉关键词：

- 内容流
- 轻学习
- 白卡片
- 浅绿色背景
- 清晰信息层级
- 温和的青少年友好感

设计原则：

- 内容优先：新闻标题、来源、时间、地点、考点和推荐信息是视觉主角。
- 轻量辅助：学习模块像新闻内容的自然延展，不像课程后台。
- 状态完整：正常、空、加载、异常、已读、生成中都要被设计。
- 模块复用：评论、音频、内容流、学习 tabs 等高复用功能必须按模块规格接入。
- 审核优先：AI 生成页面只是初稿，必须经过产品和设计审核。

不建议：

- 大面积营销式 hero。
- 过度装饰的渐变背景。
- 纯儿童化插画语言。
- 密集堆叠卡片导致阅读压力过大。
- 为单个页面临时发明新的视觉体系。

## 4. Foundations

### 4.1 颜色 Tokens

Figma 中已存在两个颜色变量集合：

- `.base_color 基础色`：基础色集合，包含原有 raw colors，并补充了 `Green 绿色 / Blue 蓝色 / Yellow 黄色 / Orange 橙色 / Red 红色 / Gray 灰色` 六组基础色盘。
- `Color_Token 颜色语义`：语义色集合，包含 `light` 和 `dark` 两个模式。

后续设计和工程应优先使用 `Color_Token 颜色语义`，不要直接引用 `.base_color 基础色`。基础色盘只作为语义色的来源。

使用规则：

- 页面背景只在 `Bg 背景/Bg_Page_Green 页面绿底`、`Bg 背景/Bg_Page_Grey 页面灰底`、`Bg 背景/Bg_Page_White 页面白底` 中选择。
- 卡片使用 `Bg 背景/Bg_Card 卡片`，浅灰内容区使用 `Bg 背景/Bg_Block 内容块`。
- 骨架和占位使用 `Bg 背景/Bg_Placeholder 占位`。
- 音频控件使用 `Bg 背景/Bg_Audioplay 音频播放`，保证和普通卡片形成明确区分。
- 绿色只用于品牌、选中、强调、正向状态，不作为大面积正文背景。
- 功能色按调用属性拆分：`Text 文字` 只用于文字颜色，`Bg 背景` 只用于填充，`Border 描边` 只用于描边；因此在 Figma 的 Fill 变量选择器中只会看到各功能色的 `Bg 背景`。
- 功能状态必须使用 `Function 功能/*` 语义色，不直接拿插画中出现的红、黄、蓝做 UI 状态色。
- 暗色模式中的品牌/功能色背景不能退化为纯灰，必须使用对应色相的深色原子，例如 `Green 绿色/950`、`Blue 蓝色/900`、`Yellow 黄色/900`、`Red 红色/900`。

### 4.2 Typography Styles

Figma 中已沉淀本地 Text Styles。字体、字号、行高不再拆成变量集合，避免变量面板过细；后续组件直接使用 Text Styles 作为字体规范来源。

字体体系以 `HYZhengYuan` 为目标字体，当前 MCP 写入环境仍无法直接加载该字体，因此 Figma 样式内保留 `Noto Sans SC` fallback，并在 description 中记录 intended 字体。

字重规则：

- `HYZhengYuan 95W`：只用于页面级标题，例如首页问候 `你好，少年！`。
- `HYZhengYuan 85W`：全局加粗体，用于新闻标题、模块标题、卡片标题、tabs、音频标题等。
- `HYZhengYuan 55W`：全局常规体，用于正文、评论、辅助信息、标签、来源时间等。

- 长文、评论、学习模块必须允许自然增长，不固定死高度。

### 4.5 State

组件状态命名：

- 通用：`default`、`selected`、`pressed`、`disabled`、`loading`、`error`。
- 内容：`read`、`empty`、`generating`。
- 列表：`hasMore`、`noMore`。
- 状态适合做组件属性和模块规范，不强行创建为 Figma string 变量。

## 5. 原子层

原子层是所有页面和模块都必须使用的基础元素。

基础元素：

- 颜色 tokens。
- 文本 styles。
- 间距 tokens。
- 圆角 tokens。
- 尺寸 tokens。
- 描边 tokens。
- 透明度 tokens。
- 投影 styles。
- 分割线。
- 图标。
- 头像。
- 标签。
- 骨架占位。
- 图标按钮。

原子层规则：

- 原子元素不承载完整业务路径。
- 原子元素必须可被组件和模块复用。
- 新增原子元素前，先确认是否能由现有 token 或基础元素表达。

## 6. 组件层

组件层是可在多个页面中复用的 UI 单元。它们可以有状态，但通常不独立承载完整业务路径。

核心组件：

| 组件 | 用途 | 关键状态 |
| --- | --- | --- |
| `TopBar` | 首页问候、详情返回/分享、个人中心头部 | `home`、`detail`、`profile`、`sticky` |
| `NewsCard` | 首页和收藏等内容列表卡片 | 默认、已读、无推荐、少推荐、加载 |
| `SectionTitle` | 内容模块标题，如少年说、对我的影响 | 默认、带操作 |
| `SegmentTabs` | 内容切换 | 默认、选中、滚动 |
| `FeatureGridItem` | 详情页学习功能入口 | 默认、选中、禁用 |
| `BottomTabBar` | MVP 一级导航 | 默认、选中、徽标 |
| `EmptyState` | 空内容提示 | 普通空、评论空、收藏空 |
| `SkeletonBlock` | 加载占位 | 卡片骨架、正文骨架 |
| `ListFooter` | 内容流到底提示 | 默认 |

设计规则：

- 组件命名要稳定，禁止使用 `Frame 13`、`Group 6` 等临时名称。
- 组件状态必须独立定义，不为每个状态创建完全不同的组件。
- 复杂业务路径不要塞进基础组件，应升级为模块。

## 7. 模块层

模块层是有完整操作路径、高复用价值、需要单独定义兼容性的功能模式。模块可以由多个组件组成，并可在不同页面中接入。

v0.1 已拆分模块：

- 评论系统：见 `modules/comment-system.md`。
- 音频控件：见 `modules/audio-player.md`。

已沉淀的 Figma 组件：

- `NewsCard`：首页信息流卡片组件集，包含 `NoRecommend`、`SingleRecommend`、`FewRecommends`、`Read`、`Loading` 5 个状态。

建议后续拆分模块：

- 新闻信息流 `news-feed`。
- 学习内容 tabs `learning-tabs`。
- PDF 生成与下载 `pdf-generation`。
- 收藏/历史列表 `saved-content-list`。

模块是否需要单独文档的判断标准：

- 是否有完整操作路径。
- 是否有多种状态和失败情况。
- 是否会在多个页面复用。
- 是否涉及数据、权限、登录、嵌入方式或跨端兼容。
- 是否需要 AI 在新增页面时稳定调用。

## 8. 核心页面模式

### 8.1 首页

页面目标：让用户快速浏览今日内容，进入新闻详情或生成简报 PDF。

结构：

1. 状态栏。
2. 首页顶部：头像/品牌形象、问候、日期、通知/任务入口。
3. 可选音频条。
4. 模块标题。
5. 可选生成 PDF 操作。
6. 新闻卡片列表。
7. 到底提示。
8. 底部导航。

必须覆盖：

- 正常列表。
- 空列表。
- 加载骨架。
- 已读卡片。
- 推荐数为 0、1、少量、多量。

### 8.2 详情页

页面目标：围绕一条新闻提供速听、解释、学习拓展、评论和素材积累。

结构：

1. 状态栏。
2. 返回与分享等顶部操作。
3. 新闻标题。
4. 来源、时间、地点、距离等元信息。
5. 音频模块。
6. 影响摘要。
7. 学习内容 tabs 或功能入口。
8. 正文与学习模块。
9. 评论系统。
10. 底部导航或底部操作区。

必须覆盖：

- 内容正常展示。
- 内容生成中。
- 不同年龄/文笔风格。
- 长文阅读。
- 评论为空和有评论。
- 上滑后的顶部吸附状态。

### 8.3 个人中心

页面目标：呈现用户学习概况和历史内容入口。

结构：

1. 状态栏。
2. 用户信息区。
3. 学习统计：学习天数、阅读文章数。
4. 内容分类：看过、评论、赞过、收藏。
5. 内容列表。
6. 空状态。
7. 底部导航。

必须覆盖：

- 有历史内容。
- 某分类为空。
- 统计数据加载中。
- 用户信息缺失或默认头像。

## 9. 状态原则

所有新增页面和模块必须考虑：

- 正常态：真实内容密度，不只放示例占位。
- 空态：说明为什么为空，并给出下一步行动。
- 加载态：使用与最终结构一致的骨架。
- 异常态：说明失败原因，提供重试或返回路径。
- 已读态：降低视觉权重，不改变列表节奏。
- 生成中：不能空白等待，应展示进度、骨架或生成提示。
- 未登录：不阻断浏览时应延迟登录，不可操作时给出登录引导。

## 10. AI 设计使用原则

AI 新增页面时必须：

- 先判断页面属于列表、详情、个人资产、学习工具、互动或生成结果。
- 优先复用底层 tokens、核心组件和已定义模块。
- 对评论、音频等复杂功能读取对应模块 spec 或 module skill。
- 输出正常、空、加载、异常至少四类状态。
- 明确说明哪些是沿用、哪些是新增。
- 附带人工审核清单。

AI 不允许：

- 从散乱画板直接复制临时结构当作新规范。
- 发明与现有视觉无关的新组件体系。
- 忽略暗色 token，但只输出浅色可用状态。
- 只设计理想态。
