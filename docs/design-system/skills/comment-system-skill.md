# Comment System Skill Draft

用途：当 AI 需要为少年博文页面新增或调整评论能力时，按本工作流执行。  
依赖：

- `../mvp-design-system-v0.1.md`
- `../modules/comment-system.md`

## 1. 触发条件

当需求包含以下任一表达时，使用本 skill：

- 增加评论区。
- 用户可以发表看法。
- 支持回复、点赞、展开回复。
- 新页面需要讨论区、互动区、留言区。
- 个人中心展示评论历史。

## 2. 先判断接入形态

AI 必须先判断评论模块属于哪种形态：

- `embedded`：内嵌在详情页或内容页底部，默认推荐。
- `drawer`：半屏抽屉，用于轻量查看。
- `standalone`：独立评论页，用于大量评论或高级筛选。
- `history`：个人中心评论历史，只展示用户评论记录，不提供完整讨论链。

默认选择：

- 新闻详情页使用 `embedded`。
- 首页卡片快速查看使用 `drawer`。
- 个人中心使用 `history`。

## 3. 必须输出的状态

无论页面多简单，必须覆盖：

- 正常态：有评论。
- 空态：无评论，展示输入引导。
- 加载态：评论骨架。
- 异常态：加载失败和重试。
- 未登录态：可浏览，不可操作时引导登录。
- 发送中/发送失败：保留用户输入。

如果支持回复，还必须覆盖：

- 有回复。
- 回复较多折叠。
- 展开更多回复。

## 4. 设计规则

必须遵守：

- 读取 `comment-system.md` 中的模块结构。
- 使用 MVP Design System 的颜色、文字、边距和状态原则。
- 评论区不抢正文主层级。
- 回复最多展示两层视觉层级。
- 首页卡片内不放完整评论区。
- 半屏抽屉必须有明确关闭路径。

禁止：

- 重新设计一套评论视觉。
- 只画正常态。
- 使用重卡片包裹每条评论。
- 把未登录状态藏到异常态里。
- 让评论输入遮挡底部导航或安全区。

## 5. 输出格式

AI 生成评论模块方案时，输出：

```md
## 评论模块接入方案

接入形态：
放置位置：
用户路径：
组件结构：
状态覆盖：
数据需求：
视觉约束：
交互说明：
异常与权限：
人工审核清单：
```

## 6. 接入模板

### 新闻详情页

```txt
ArticleDetailPage
  ArticleContent
  LearningModules
  CommentSection(embedded)
```

规则：

- 评论区放在正文和学习模块之后。
- 顶部先展示输入引导，再展示评论列表。
- 评论为空时仍保留表达入口。

### 首页卡片快速评论

```txt
NewsCard
  CommentCount
  OpenCommentDrawer
    CommentSection(drawer)
```

规则：

- 卡片只展示评论入口或计数。
- 抽屉不承载复杂排序和深层回复。

### 个人中心评论历史

```txt
ProfilePage
  ProfileContentTabs
  CommentHistoryList
```

规则：

- 展示用户评论过的内容和原新闻入口。
- 不展示完整回复链，必要时跳转详情页。

## 7. 审核清单

设计：

- 状态是否完整。
- 评论层级是否清楚。
- 输入区是否和底部导航兼容。

产品：

- 页面是否真的需要评论能力。
- 未登录、删除、举报、审核是否有明确策略。

工程：

- 是否可复用 `CommentSection`。
- 页面是否只负责传入目标对象和权限，不内写评论逻辑。
