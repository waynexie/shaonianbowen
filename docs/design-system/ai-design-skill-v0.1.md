# 少年博文 AI Design Skill v0.1

用途：指导 AI 基于 MVP Design System v0.1 生成二期新增页面的设计初稿。  
适用对象：产品、设计、AI agent、前端实现前的页面方案生成。  
输入依赖：`mvp-design-system-v0.1.md`、Figma 文件 `少年博文` 的 `给开发` 页面。

模块依赖：

- 页面需要评论能力时，读取 `modules/comment-system.md`；需要可执行流程时，读取 `skills/comment-system-skill.md`。
- 页面需要音频播放、听全文、速听、跟读时，读取 `modules/audio-player.md`；需要可执行流程时，读取 `skills/audio-player-skill.md`。

## 1. 使用边界

AI 可以做：

- 基于已有首页、详情页、个人中心模式生成新增页面初稿。
- 组合已有组件生成正常态、空态、加载态、异常态。
- 在复杂功能中调用模块规格，例如评论系统、音频控件。
- 产出页面结构、组件清单、状态说明和人工审核清单。
- 在明确要求时，生成 Figma 页面草案或前端实现建议。

AI 不应做：

- 未经审核直接修改正式 Figma 组件库。
- 发明全新视觉语言替代现有绿色、卡片、内容流风格。
- 直接复制散乱 Frame 当作新规范。
- 在未确认工程组件存在时假设已经有成熟代码组件。

## 2. 输入模板

新增页面需求必须尽量包含：

```md
页面名称：
页面目标：
目标用户：
入口：
出口：
核心内容：
主要操作：
数据状态：
是否需要音频/评论/PDF/收藏：
是否涉及年龄段内容：
成功标准：
```

如果用户没有提供完整信息，AI 使用以下默认值：

- 画板尺寸：移动端 `390px` 宽。
- 页面边距：左右 `20px`。
- 视觉风格：沿用 MVP 绿色、白卡、内容流、学习辅助模块。
- 状态覆盖：正常态、空态、加载态、异常态。
- 组件复用：优先使用 v0.1 组件，不新增视觉体系。
- 模块复用：评论、音频等复杂功能优先调用模块规格，不临场重画。

## 3. 生成流程

### Step 1: 识别页面类型

先判断新增页面属于哪一类：

- 内容列表类：参考首页信息流。
- 内容详情类：参考详情页底层/上滑态。
- 用户资产类：参考个人中心。
- 学习工具类：参考详情页 tabs、作文金句、考点分析、中英对照。
- 互动类：参考评论区、发表看法、回复展开。
- 生成结果类：参考 PDF 生成、内容生成态、骨架态。

如果页面涉及评论或音频，先进入对应 module skill，再回到本流程组合页面。

### Step 2: 选择页面模式

必须从以下模式中选择或组合：

- `HomeFeedPattern`：顶部问候、音频条、模块标题、内容卡片列表。
- `ArticleDetailPattern`：标题、来源、音频、影响摘要、tabs、正文、评论。
- `ProfilePattern`：用户信息、学习统计、分类 tabs、历史列表。
- `CommentPattern`：输入引导、评论列表、回复展开、空态。
- `LearningModulePattern`：考点、金句、观点、中英对照、跟读。

### Step 3: 选择组件

优先使用：

- `TopBar`
- `NewsCard`
- `AudioPlayerBar`
- `CommentSection`
- `SectionTitle`
- `SegmentTabs`
- `FeatureGridItem`
- `CommentItem`
- `EmptyPlaceholder`
- `SkeletonCard`
- `BottomTabBar`
- `Footer`

只有当以上组件不能表达页面目标时，才允许新增组件。新增组件必须说明：

- 为什么现有组件不够。
- 新组件服务哪个具体任务。
- 它与现有组件的关系。
- 至少包含哪些状态。

### Step 4: 输出页面设计说明

AI 输出必须包含：

- 页面目标。
- 信息架构。
- 首屏结构。
- 组件清单。
- 状态清单。
- 交互说明。
- 文案草案。
- 人工审核清单。

如果生成 Figma 页面，还需要说明：

- 参考了哪些原始画板或组件。
- 哪些部分是沿用。
- 哪些部分是新增。
- 哪些部分需要设计师二次整理成正式组件。

## 4. 视觉约束

布局：

- 默认移动端宽度使用 `Size 尺寸/Page 页面/MobileWidth 移动画板宽`。
- 页面左右边距使用 `Size 尺寸/Page 页面/PaddingX 左右边距`。
- 主内容宽度使用 `Size 尺寸/Content 内容/MobileWidth 主内容宽`。
- 底部导航高度使用 `Size 尺寸/BottomTab 底部导航/Height 高度`。
- 状态栏高度使用 `Size 尺寸/StatusBar 状态栏/Height 高度`。
- 图标按钮视觉尺寸使用 `Size 尺寸/IconButton 图标按钮/Size 尺寸`，工程点击热区不小于 `Size 尺寸/TouchTarget 点击热区/Min 最小`。

颜色：

- 背景优先使用 `Bg 背景/Bg_Page_Green 页面绿底`、`Bg 背景/Bg_Page_Grey 页面灰底`、`Bg 背景/Bg_Page_White 页面白底`。
- 卡片使用 `Bg 背景/Bg_Card 卡片`。
- 音频条使用 `Bg 背景/Bg_Audioplay 音频播放`。
- 骨架使用 `Bg 背景/Bg_Placeholder 占位`。
- 主文本使用 `Text 文本/T1 一级文本`。
- 辅助文本使用 `Text 文本/T2 二级文本` 或 `Text 文本/T3 三级文本`。
- 绿色强调使用 `Color 彩色/Standard_Green 标准绿` 或 `Text 文本/Text_Green 绿色文本`。

文字：

- 新闻标题优先保留 2 行可读性。
- 详情页正文行高必须舒适，不要压缩成资讯摘要密度。
- 空态文案轻松但清楚。
- 错误态文案明确原因和下一步动作。
- 字号、行高、字重优先使用本地 Text Styles。

交互与投影：

- 禁用、已读、按下、遮罩和骨架透明度使用 `Opacity 透明度`。
- 默认不使用投影；确有浮起关系时使用 `DS/Shadow/*` Effect Styles。
- 底部栏使用 `DS/Shadow/BottomBar 底部栏`，弹出菜单使用 `DS/Shadow/PopoverMenu 弹出菜单`。
- 动效和层级暂不作为 Figma 变量使用，AI 设计页面时只描述交互意图，不发明新的 token。

组件：

- 不把卡片套卡片。
- 不使用与现有稿无关的大面积渐变装饰。
- 不将所有内容都做成卡片，页面分区可使用背景和留白。
- 不为图标按钮写长文本，图标需配 hover/无障碍名称或工程语义。

## 5. 状态生成要求

每个新增页面至少输出 4 个状态：

| 状态 | 必须说明 |
| --- | --- |
| 正常态 | 用户最常见路径，包含真实内容密度 |
| 空态 | 为什么为空，下一步可做什么 |
| 加载态 | 哪些区域使用骨架，是否保留布局高度 |
| 异常态 | 失败原因、重试或返回路径 |

与内容相关页面还需考虑：

- 已读 / 未读。
- 收藏 / 未收藏。
- 点赞或推荐数量为 0、1、少量、多量。
- 评论为空、有评论、有回复展开。
- 内容生成中、生成失败、生成成功。

## 6. 页面示例策略

### 搜索页

推荐组合：

- `TopBar` 或搜索头部。
- 搜索输入。
- 热门考点/历史搜索。
- `NewsCard` 列表。
- 空结果 `EmptyPlaceholder`。
- 加载 `SkeletonCard`。

### 收藏页

推荐组合：

- `TopBar`
- `SegmentTabs`：文章、金句、评论或 PDF。
- `NewsCard` 或收藏条目。
- `EmptyPlaceholder`
- `BottomTabBar`

### 设置页

推荐组合：

- `TopBar`
- 分组列表。
- 用户/阅读偏好入口。
- 文笔年龄段设置。
- 通知、隐私、账号相关操作。

### 发布/评论页

推荐组合：

- `TopBar`
- 输入区域。
- 当前新闻上下文摘要。
- 发布按钮。
- 字数/情绪提示。
- 失败重试状态。

## 7. 输出格式

AI 设计页面时，统一使用以下格式：

```md
# 页面名称

## 目标

## 信息架构

## 首屏布局

## 组件清单

## 状态设计

## 交互说明

## 文案草案

## 复用与新增

## 人工审核清单
```

## 8. 人工审核清单

设计审核：

- 是否符合少年博文的内容学习产品定位。
- 是否沿用已有绿色、卡片、内容流、学习模块语言。
- 是否覆盖正常、空、加载、异常状态。
- 是否没有引入不必要的新组件。
- 是否移动端首屏信息层级清楚。

产品审核：

- 页面目标是否单一明确。
- 入口和出口是否闭环。
- 空态和异常态是否给出下一步动作。
- 文案是否适合青少年及其家长/教育场景。
- 重要内容是否没有被弱化或折叠过深。

工程审核：

- 是否能映射到既有或计划中的组件。
- 是否使用语义 token，而不是硬编码颜色。
- 是否避免固定死长文高度。
- 是否考虑 loading、error、empty 的数据状态。
- 是否能在 390px 宽度和常见手机高度下正常展示。
